# מערכת SaaS ליצירת תאומים דיגיטליים אינטראקטיביים

## תקציר והגדרת הצלחה

המערכת תהיה פלטפורמת SaaS מרובת לקוחות, המייצרת מודל מוצר אינטראקטיבי מתמונות, מידות, מסמכים וקובצי CAD אופציונליים.

התוצר יכלול:

- חוויית Web אינטראקטיבית המתאימה למחשב ולמובייל.
- קובצי `GLB`, `USDZ`, קובץ Blender ומפות PBR.
- חלקים פעילים: כפתורים, מתגים, מסכים, דלתות, גלגלים ומנגנונים.
- עורך חזותי לתיקון המודל, ההתנהגות, החומרים והתאורה.
- מצב Blender למשימות מידול מתקדמות.
- פרסום לעמוד מוצר, קוד הטמעה או הורדת הנכסים.

לא ניתן להבטיח דיוק הנדסי מתמונות בלבד. לכן יהיו שני מצבים:

- **Commercial Mode:** נאמנות חזותית גבוהה וחוויית מוצר מסחרית.
- **Engineering Mode:** דורש CAD, סריקה או מידות עם טולרנסים מוגדרים.

## זרימת העבודה

1. המשתמש יוצר פרויקט, בוחר סוג מוצר, מצב איכות ומדיניות פרטיות.
2. מעלה תמונות מכל הזוויות, מידות, הוראות, לוגו, CAD וסרטוני פעולה.
3. המערכת בודקת חדות, כיסוי זוויות, עקביות ומידע חסר ומבקשת השלמות במידת הצורך.
4. AI מנתח חלקים, חומרים, מסכים, כפתורים והתנהגויות אפשריות.
5. נוצרים מספר מועמדי 3D באמצעות ספקים ושיטות שונות.
6. מנוע הערכה משווה את המועמדים למקורות ובוחר את התוצאה הטובה ביותר.
7. עובדי Blender אוטומטיים מתקנים גאומטריה, קנה מידה, טופולוגיה, UV, חומרים ופיבוטים.
8. המערכת מייצרת אינטראקציות ופיזיקה מתוך מפרט המוצר.
9. מתבצעות בדיקות חזותיות, פונקציונליות וביצועים.
10. המשתמש מקבל תוצאה מלאה ועורך תיקונים בעורך החזותי או ב-Blender.
11. גרסה מאושרת מתפרסמת ומתווספת למאגר הלמידה המאושר.

## ארכיטקטורת המערכת

- **Frontend:** Next.js, React, Three.js ו-React Three Fiber.
- **עורך 3D:** בחירת חלקים, שינוי חומרים, פיבוטים, תאורה, אנימציות ומצבי מוצר.
- **Control Plane:** שירותי FastAPI לניהול פרויקטים, ספקי AI, יצירה והערכה.
- **Workflow Engine:** Temporal לניהול משימות ארוכות, ניסיונות חוזרים והמשך לאחר כשל.
- **Workers:** קונטיינרים של Blender במצב Headless ועובדי GPU.
- **נתונים:** PostgreSQL עם הפרדה מלאה בין לקוחות, Redis לתורים ומכסות ואחסון S3-compatible לנכסים.
- **אבטחה:** Vault/KMS למפתחות API, כתובות הורדה חתומות, הצפנה, RBAC, Audit Log ו-SSO.
- **פריסה:** Kubernetes עם Workers בענן ואפשרות ל-Private Worker ברשת הלקוח.

המודל הקיים של BeeTech ישמש כפרויקט Golden Reference לבדיקת איכות האינטראקציות, המסך, המתגים והתצוגה במובייל.

## מפרט התאום הדיגיטלי

כל מוצר יישמר כ-`ProductDigitalTwinSpec` מובנה:

- זהות מוצר, קטגוריה, מידות, יחידות ומצב דיוק.
- היררכיית חלקים, שמות, מיקום, פיבוט ויחסי הורה-ילד.
- גאומטריה, חומרים, מרקמים, לוגואים ומאפייני PBR.
- אזורי לחיצה, Hotspots וחלקים ניתנים להזזה.
- מסכים, טקסטים, תאורה, שמע ומצבי תצוגה.
- State Machine המתאר מצבי מוצר ומעברים.
- מפרט פיזיקה, התנגשויות ומגבלות תנועה.
- רמת ביטחון ומקורות ההוכחה לכל החלטת AI.

## מנוע אינטראקציה ופיזיקה

המקור הקובע להתנהגות יהיה State Machine דטרמיניסטי. מנוע Rapier יופעל רק כאשר נדרשת סימולציה פיזיקלית אמיתית.

לכל חלק פעיל יוגדרו:

- מערכת צירים מקומית, יחידות במטרים, מסה ומרכז כובד.
- Collider פשוט לביצועים ו-Mesh נפרד לתצוגה.
- חיכוך, קפיציות, התנגדות והתנגשות.
- סוג מפרק: `fixed`, `revolute`, `prismatic`, `spherical` או מנגנון מותאם.
- ציר תנועה, מגבלות, מרחק, זווית, קפיץ, damping ונקודת snap.
- אירועים: לחיצה, שחרור, גרירה, סיבוב, הפעלה וכיבוי.
- השפעות: אנימציה, שינוי מסך, תאורה, שמע ומעבר מצב.

דוגמאות:

- כפתור יהיה מפרק ליניארי קצר עם חזרה קפיצית ואירוע בנקודת הלחיצה.
- מתג OFF/ON יהיה מפרק ליניארי או סיבובי עם שתי נקודות snap.
- דלת או מכסה יהיו מפרק סיבובי עם מגבלות זווית.
- מסכים יפעלו לפי State Machine ולא כסימולציית פיזיקה.
- בדים, נוזלים וחומרים רכים יטופלו בחבילות יכולת נפרדות, לרוב באמצעות אנימציה אפויה ולא סימולציה מלאה בדפדפן.

## ניהול ספקי AI ומפתחות

ייבנה `ProviderAdapter` אחיד לכל ספק. לא ניתן לחבר API שרירותי ללא Adapter המתאר את היכולות, מבנה הבקשה, התוצאה, העלויות ותנאי השימוש שלו.

הנתב יתחשב ב:

- סוג היכולת הנדרשת: Vision, Multi-Image-to-3D, Texture, Segmentation או Reasoning.
- רף איכות, הרשאות פרטיות, מחיר, זמינות, זמן תגובה ומכסה.
- שגיאות `401`, `402`, `429` ו-`5xx`.
- Retry, Circuit Breaker, Cooldown ומעבר לספק תואם.
- מפתחות מערכת ומפתחות BYOK מוצפנים.
- תקציב יומי, חודשי ולפי פרויקט.

המערכת לא תעבור לספק חלש יותר רק כדי להמשיך לעבוד. אם אין ספק חלופי שעומד ברף האיכות והפרטיות, המשימה תמתין ותדווח למשתמש.

## למידה ושיפור עצמי

“למידה עצמית” לא תהיה שינוי אוטונומי ולא מבוקר של המערכת.

המערכת תלמד רק מתוצרים ותיקונים שאושרו:

- שמירת תיקוני משתמש כמקרי הצלחה לפי קטגוריית מוצר.
- שימוש בפרויקטים דומים כדי לבחור תבניות, ספקים ופרמטרים.
- דירוג ביצועי ספקים לפי סוג משימה.
- שיפור Prompts, תהליכים וחוקי הערכה בגרסאות מבוקרות.
- ניסויי Champion/Challenger לפני החלפת תהליך קיים.
- Fine-tuning רק על Dataset מאושר ועם אפשרות Rollback.
- בידוד מלא בין נתוני לקוחות ואיסור אימון ללא הסכמה מפורשת.

## ממשקים מרכזיים

- `POST /projects` — יצירת פרויקט ומדיניות פרטיות.
- `POST /projects/{id}/assets` — העלאת תמונות, CAD, מידות ומסמכים.
- `POST /projects/{id}/generate` — התחלת Pipeline לפי מצב איכות ותקציב.
- `GET /jobs/{id}` — מצב, עלויות, ספק פעיל ותוצאות ביניים.
- `PATCH /digital-twins/{id}` — שמירת תיקוני העורך החזותי.
- `POST /digital-twins/{id}/validate` — בדיקות איכות ופיזיקה.
- `POST /digital-twins/{id}/publish` — יצירת חבילת Web וקובצי הפצה.
- `POST /provider-credentials` — הוספת מפתח BYOK מוצפן.

## איכות, בדיקות ותנאי קבלה

- השוואת Render אוטומטית לכל זוויות המקור.
- בדיקות מידות וטולרנסים במצב Engineering.
- בדיקת glTF Validator ו-Asset Auditor לכל תוצר.
- בדיקות State Machine לכל מעבר ומצב.
- הרצת אותה אינטראקציה מספר פעמים וקבלת תוצאה זהה.
- בדיקת Failover כאשר ספק נכשל, נגמרים הקרדיטים או חורגים מהמכסה.
- בדיקת בידוד לקוחות, מפתחות והרשאות.
- יעד ביצועים: לפחות 30FPS במובייל ו-60FPS במחשב נתמך.
- LOD, דחיסת גאומטריה וטקסטורות KTX2 בהתאם למכשיר.
- אין פרסום אוטומטי כאשר קיימת שגיאת איכות קריטית.

## שלבי פיתוח

1. **מחקר ו-Benchmark, כ-6 שבועות:** בניית מאגר מוצרי בדיקה והשוואת ספקי AI.
2. **Commercial MVP, כ-4–6 חודשים:** אלקטרוניקה ומוצרים קשיחים, עורך חזותי, Blender Workers ופרסום Web.
3. **SaaS Production, כ-3–4 חודשים:** לקוחות מרובים, BYOK, חיוב, הרשאות, ניטור ו-Private Workers.
4. **Engineering Mode וחבילות נוספות, כ-4–6 חודשים:** CAD, טולרנסים, אריזות וריהוט.
5. **Soft Goods ואורגני, כ-6–12 חודשים:** ביגוד, מוצרים רכים ואינטראקציות מורכבות.

למערכת בהיקף של 100–1,000 מוצרים בחודש נדרש צוות התחלתי של 6–10 אנשים: פיתוח Frontend/3D, Backend, ML/CV, Technical Artist ל-Blender, DevOps, QA ומוצר.

## הנחות שנקבעו

- המערכת תהיה SaaS מרובת לקוחות עם Web UI ועובדי עיבוד היברידיים.
- כל קטגוריות המוצרים ייתמכו בהדרגה באמצעות חבילות יכולת.
- איכות קודמת לרציפות ולעלות.
- התמונות יגיעו עם מידות, ו-CAD ישמש כאשר קיים.
- המשתמש יקבל תוצאה מלאה ואז יתקן אותה.
- נתונים יישלחו לספקים לפי מדיניות פרטיות של כל פרויקט.
- מפתחות API יישמרו רק בכספת שרת מוצפנת.
